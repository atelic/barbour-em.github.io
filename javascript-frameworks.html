Comparing JavaScript Frameworks

Posted on Wed 08 June 2016 in Javascript

Motivation

There are an unbelievable amount of JavaScript frameworks in the world and it seems like there is a new one every week. At Fedora, we are evaluating adding a JavaScript framework to some of our applications, especially Fedora Hubs. This is new for most of the team members as almost all Fedora apps use server side templating with little to no client-side templating. Since few team members have a preference on which framework to use and the desire to avoid picking one at random, I have compiled some notes on some of the most popular JS frameworks and ones that I have used in the past. These are my biased opinions either based on my experience with the framework or from reasearch I have done on them.

Overview of various JavaScript frameworks

AngularJS

AngularJs is an extremely popular framework and has been for some time. This makes it a pretty safe choice as it is likely that this popularity will continue. The newest version (2.0 just released from beta) represents a huge change over the first version. Angular 2.0 is written in TypeScript, which adds type checking and other enhancements to JavaScript. It also attempts to make developing for mobile easier and improves its initial router.

The issue with choosing Angular at the moment is the strange limbo it seems to be in. Version 2.0 is very new and contains breaking changes but version 1.x will soon be obsolete. This is compounded by the difficulty associated with upgrading Angular applications from a previous version. Solutions are available, it is a major overhaul to be done.

Angular2 Example source

A simple store of hard-coded todo items:

export class TodoItem {
  text: String;
}

export class TodoStore {
  items: TodoItem[] = [
     {text: 'Feed the cat'},
     {text: 'Do the shopping'}
  ];
}

And the associated template:

<template ngFor #item [ngForOf]="items">
  <li>
    <div class="view">
      <label></label>
    </div>
  </li>
</template>

Pros

  • Nice (less noisy) templating system
  • Large ecosystem as it's probably the most popular.
    • This chart shows just how popular Angular is with developers compared to other major JavaScript frameworks.
  • A lot of functionality out of the box without the need for libs
  • Typescript

Cons

  • Learning curve
    • Quick to get started, then it improves and deteriorates as you go along
  • A bit of a leaky abstraction (i.e. have to understand underlying model to understand the abstraction)
  • New version coming out right now
    • Start with new one (in beta) or old one (that will be depreciated)
  • Long list of Angular-specific syntax
  • New version has some lacking documentation

Backbone

Backbone.js is a JavaScript library that gives structure and organization to JavaScript applications. It is small and unopinionated, which, as a result, means that it may not do everything that you need right out of the box. One of the issues is that while it allows you to organize your application code, it makes no attempt at view templating, so another library is needed.

Example source

app.Todos = Backbone.Collection.extend({
    // Reference to this collection's model.
    model: app.Todo,

    completed: function () {
        return this.where({completed: true});
    },

    remaining: function () {
        return this.where({completed: false});
    },

    nextOrder: function () {
        return this.length ? this.last().get('order') + 1 : 1;
    },

    comparator: 'order'
});
// The associated model
app.Todo = Backbone.Model.extend({
    // Default attributes for the todo
    defaults: {
        title: '',
        completed: false
    },
    toggle: function () {
        this.save({
            completed: !this.get('completed')
        });
    }
});

Pros

  • Small and fast
  • Gives the developer absolute control over what other components of the application to add or leave out

Cons

  • No data binding
  • A number of dependencies depending on if you want:
    • routing
    • templating
    • data binding
    • async events
    • underscore

React

React descibes itself as "A JavaScript library for building user interfaces," which basically means it is just the V in MVC. It provides the view layer only and "makes no assumptions about the rest of your technology stack."

React has steadily been rising in popularity over the past year or two and there has grown a large following and ecosystem of plugins around it. A notable example of this is Flux which provides the application architecture layer to hook into React's view layer. React feels like a good choice for new or existing projects. Its modular nature makes it easier than others to begin implementing new UI in React or slowly moving other elements over.

Example (with JSX) source

var HelloMessage = React.createClass({
  render: function() {
    return <div>Hello {Page on this.props.name}</div>;
  }
});
React.renderComponent(<HelloMessage name="John" />, mountNode);

Pros

  • JSX - HTML-like syntax that compiles to JS
    • Which means errors at compile time itself, rather than on runtime like Angular
  • Lightweight and a simple API. Don't need to read source code to understand what's happening
  • Most of it is just plain JS or close to it. Less doc reading
  • Binding markup doesn't changed bases on one vs two way
    • Which feels more JavaScripty
  • ECMAScript 6 compatable

Cons

  • Library not a framework (no router nor model management)
    • Pick and choose what you want to use with it (can be daunting)
    • One needs to have some degree of experience in picking libraries to fulfill these needs.
  • Learning curve
    • A bit difficult to pick up at first
  • Not as many developers with experiences as a whole

Knockout

Knockout is a library heavily focused on data binding. It is not an MVC framework per se, but idiomatic Knockout code uses the similar concept of view models, which are a combination of model and controller layers in a single object. Its templates are written in HTML with data-bind attributes similar to that of Angular; this has the disadvantage of making debugging difficult at times but the advantage of sophisticated and nested UIs.

However, Knockout is only for two-way data binding, it automatically refreshes the UI as the data changes. This can lead to severe performance issues if the developer doesn't entirely grok the way Knockout does UI refreshes. It this means that if hash based routing or UI components are needed, another library will have to be included.

Example

   <html>
        <head>
            <script type="text/javascript" src="knockout.js"></script>
        </head>
        <body>
            <span data-bind="text: hello"></span>
            <script type="text/javascript">
                var viewModel = function() {
                    this.hello = ko.observable("Hello, World!");
                }
                ko.applyBindings(viewModel;
            </script>
        </body>
    </html>

Pros

  • Lightweight (20kb with HTTP compression)
  • The easiest on this list to pick up in my opinion with good documentation and interactive tutorial system.
  • Declarative two-way binding enables things like UI events propagation, visible true/false and dynamic UI.
  • Dependencies are handled through a dependency graph which targets specific data as opposed to updating entire models when data changes

Cons

  • Not a full MVC framework, but more of a versitile data binding library.
  • Can have performance issues if code isn't written with underlying implemtation in mind.
  • All javascript variables and arrays are functions.
  • HTML views can get messy as declarative bindings increase

Mithril

Mithril is a full blown client-side MVC framework that uses virtual DOM diffing similar to that of React, compilable templates, and a smart auto redraw system. The performance numbers on its web page are very flattering as well. Mithril is interesting because it has a very small api with only 24 built-in functions, and yet covers the same range as Angular in terms of MVC. While it doesn't have a large user base, it has been used on sites like melpa.org.

Example source

app.viewName = function(ctrl, task) {
    return m('li', { class: task.completed() ? 'completed' : ''}, [
        m('.view', [
            m('input.toggle[type=checkbox]', {
                onclick: m.withAttr('checked', task.completed),
                checked: task.completed()
            }),
            m('label', task.title()),
            m('button.destroy', { onclick: ctrl.remove.bind(ctrl, index)})
        ]),
        m('input.edit')
    ]);
};

Pros

  • Extremely small API, learning curve, and file size (7.8 kB!)
  • Fast performance with virtual DOM diffing
  • Flexibe in implementation
  • Its uncompiled templates run natively in the browser as compared to React's which do not

Cons

  • Small user base
  • Difficult to scale - code can become spaghetti easily if not careful.
  • Not flexible - render a Virtual DOM and assign event handlers with a few nice methods for common patterns like HTTP Requests and Promises
  • Frankly ugly templating

Honorable mentions

It is imposible for me to have touched on all of the frameworks out there so here is a list of MVC frameworks or libraries that I didn't have time to touch on but have heard good things about:

  • Aurelia
  • Ember
  • Meteor
  • Polymer
  • Riot
  • Vue

Conclusions

So, after going over all of these frameworks, it seems apparent that the one that will work best for you largely depends on the type of project that you are working on. If you just want to better organize your jQuery application, Backbone might be what you need. If you are spinning up a small app or don't need to worry about other developers learning the framework, it might be best to use Knockout or Mithril since you can get started with them very quickly. For large or enterpise applications with a large number of developers, I think either Angular or React+[Flux|Redux] is the best choice. Additionally, while the jury is still out on Angular2, it might be best to evaluate the the changes made in the new version and if you want to take the risk.
